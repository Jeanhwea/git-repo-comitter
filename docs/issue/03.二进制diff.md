# 二进制 diff 优化

## 背景

 当前 `grc` 在获取 Git 差异时，执行 `git diff --staged` 或 `git diff`，将输出直接传递给 LLM 生成提交信息。但 `git diff` 对二进制文件（图片、PDF、编译产物等）会输出 `Binary files a/X and b/X differ` 这样的占位行，其中：

 - **无实质 diff 内容**，浪费 LLM 的 token 预算
 - **无法被 `generateCommitMessageBatched` 的 `parseDiffBlocks` 正确解析**：该函数按 `diff --git a/X b/X` 行分割 block，但二进制 diff 行的格式不同，会破坏分组逻辑
 - **占用 `maxInputTokens` 额度**，可能触发不必要的分批流程

 需要在 diff 获取后、LLM 调用前，过滤或跳过二进制文件的 diff 输出。

## 方案

### 方案 A：在 Git diff 执行时排除二进制文件（推荐）

 修改 `src/git/diff.ts` 中的 diff 获取逻辑，在 `git diff` 命令中添加 `--binary` 的反向选项：

 ```bash
 git diff --diff-filter=d "$@"  # 仅排除 deleted 文件；二进制文件仍会产生 "Binary files differ"
 ```

 更精确的方式是追加 `-G"."` 参数，只输出包含文本变更的 diff，完全跳过纯二进制文件：

 ```bash
 git diff --staged -G"." -- . 2>/dev/null || git diff --staged
 ```

 `-G"."` 的含义：只显示匹配正则 `.`（即至少一个字符）的变更文件的补丁。二进制文件没有文本 diff 内容，因此 `-G"."` 会直接跳过它们。

 **优点**：
 - 一层改动，不涉及解析逻辑
 - 二进制文件不会被计入 token 估算，不会触发不必要的分批
 - `parseDiffBlocks` 不会收到它无法解析的畸形片段

 **缺点**：
 - `git diff --staged -G"."` 当所有变更都是二进制文件时，输出为空字符串 → 调用方需处理"无 diff 可提交"的分支

### 方案 B：在 diff 内容中过滤二进制 block

 在 `src/llm/split.ts` 的 `parseDiffBlocks` 中，或新增一个预处理步骤，过滤掉二进制 diff block：

 ```typescript
 // 二进制 diff 行匹配：Binary files a/X and b/X differ
 const BINARY_DIFF_RE = /^Binary files .+ differ$/;

 // 判断一个 block 是否应被丢弃
 function isBinaryBlock(block: string): boolean {
   return BINARY_DIFF_RE.test(block.trim());
 }
 ```

 在 `parseDiffBlocks` 中，分割 block 后过滤掉 `isBinaryBlock` 为 true 的条目。

 **优点**：
 - 对 Git diff 命令零改动
 - 过滤逻辑与 diff 获取解耦

 **缺点**：
 - 需要额外解析一次（虽然 token 级代价很低）
 - 二进制 diff 仍计入 token 估算，可能触发不必要的分批流程
 - `git diff` 的 `--diff-algorithm` 在某些边缘情况会产生其他非标准格式，需要额外维护

## 决策

 **采用方案 A**。原因：

 1. `-G"."` 是 Git 原生的过滤机制，性能更高
 2. 减少一层解析负担，LLM 调用层不会收到无效输入
 3. 与现有 `parseDiffBlocks` 的分割逻辑（按 `diff --git` 行）完全兼容，不产生畸形 block
 4. 实现代价极低——修改 `src/git/diff.ts` 中构建参数列表的一行

 在 `src/git/diff.ts` 中，将 `args` 构建逻辑改为：

 ```typescript
 // 当前
 const args = [diffMode === 'staged' ? '--staged' : ''].filter(Boolean);
 // 改为
 const args = [
   diffMode === 'staged' ? '--staged' : '',
   '-G"."',
 ].filter(Boolean);
 ```

 对于空 diff 的情况（所有变更都是二进制文件），调用方 `src/commands/commit.ts` 中已有针对空 diff 的处理逻辑，只需确保该分支输出"无可提交的变更"后正常退出，而非报错。

## 副作用

 - 纯二进制变更的仓库执行 `grc` 时，输出"没有文本变更可供提交"，不调用 LLM，不消耗 token。
 - 混合变更（文本 + 二进制）只对文本变更生成提交信息，二进制文件会被 `git commit` 正常提交（因为暂存区中已包含二进制文件的 blob），但提交信息中不会提及它们。这是一个可接受的折中——提交信息描述"变更意图"，而非文件清单。

## 验证

 1. 纯文本变更：行为不变，diff 正常传递
 2. 纯二进制变更（如图片替换）：`grc` 输出"没有文本变更可供提交"并退出
 3. 混合变更：LLM 得到只包含文本文件的 diff，提交信息准确反映文本变更
 4. `pnpm check && pnpm lint && pnpm build` 无报错
 5. `grc -s`（仅暂存区）在二进制文件暂存时同上行为
